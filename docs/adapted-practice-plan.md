# Адаптированный план практики

## Общая тема

Тема производственной практики формулируется как разработка и апробация автоматизированной DevOps benchmark-платформы для оценки влияния контейнеризации на развертывание, масштабирование и эксплуатационные характеристики микросервисных приложений.

Практическая часть связана с темой ВКР: «Исследование влияния контейнеризации на эффективность развертывания и масштабируемость микросервисных архитектур». В проекте реализована учебная платформа, которая объединяет frontend-dashboard, backend API, PostgreSQL, benchmark engine, Docker Compose, benchmark-target сервис, CI/CD workflow, базовый Terraform-слой и документацию.

План адаптирован под фактический результат: проект не сводится только к отдельному эксперименту VM vs Docker, а представляет собой инструментальную платформу для запуска, хранения, визуализации и автоматической проверки benchmark-сценариев.

## Связь с НИР1

НИР1 был посвящен теоретическому исследованию виртуальных машин, контейнеризации, DevOps-подходов и микросервисной архитектуры. Текущий проект переводит эти положения в прикладную архитектуру:

- выделены независимые сервисы `frontend`, `backend`, `postgres` и `benchmark-target`;
- используется контейнерная упаковка компонентов через Dockerfile и Docker Compose;
- реализован DevOps-контур локального запуска, сборки и автоматической проверки;
- зафиксированы роли инфраструктурных компонентов и ограничения учебной платформы в документации.

Таким образом, результаты НИР1 использованы как теоретическая основа для выбора архитектуры и технологического стека платформы.

## Связь с НИР2

НИР2 сформировал критерии сравнительной оценки инфраструктурных подходов: lead time, deployment frequency, MTTR, эксплуатационная сложность, масштабируемость. В текущем проекте эти критерии адаптированы к реализуемым и наблюдаемым метрикам платформы:

- `leadTimeSeconds` отражает скорость доставки изменений в демонстрационной модели;
- `scalingTimeSeconds`, `startupTimeSeconds` и `restartTimeSeconds` используются для оценки масштабирования и восстановления контейнерного окружения;
- `rps`, `latencyMs`, `p95LatencyMs`, `errorRatePercent` и `totalRequests` используются для оценки производительности под нагрузкой;
- история benchmark-запусков позволяет сравнивать результаты во времени;
- recommendation logic и comparison table интерпретируют различия между VM baseline и Docker.

Часть критериев НИР2 реализована как прямые метрики, часть представлена через методологическое описание и ограничения. Это соответствует исследовательской задаче, поскольку платформа фиксирует как измеряемые показатели, так и границы их применимости.

## Связь с НИР3

НИР3 включал натурный эксперимент VM vs Docker с использованием Terraform, облачной инфраструктуры, benchmark-нагрузки и метрик Time to Provision, Lead Time, Scaling Time, RPS, Latency, CPU Overhead.

Текущий проект продолжает НИР3 следующим образом:

- сохраняет сравнение `vm` и `docker` как два типа окружения;
- реализует benchmark-нагрузку через k6 и отдельный `benchmark-target` сервис;
- сохраняет результаты экспериментов в PostgreSQL;
- визуализирует историю, тренды и сравнение VM vs Docker во frontend-dashboard;
- включает Docker runtime metrics для контейнерного сценария;
- содержит Terraform-каталог как учебную IaC-заготовку для дальнейшего cloud deployment.

В отличие от НИР3, текущая реализация делает акцент не только на разовом эксперименте, а на повторяемой benchmark-платформе с API, историей запусков, dashboard и CI/CD-проверкой.

## Этап 1

### Название этапа

Анализ требований и уточнение архитектуры DevOps benchmark-платформы.

### Суть работ

На первом этапе была уточнена постановка практической задачи: вместо разработки разового benchmark-скрипта сформирована архитектура платформы, обеспечивающей запуск benchmark-сценариев, хранение результатов, визуализацию метрик и проверку работоспособности через DevOps-инструменты.

Были определены основные компоненты системы: React frontend-dashboard, NestJS backend API, PostgreSQL, benchmark engine, benchmark-target сервис, Docker Compose, CI/CD workflow и Terraform-заготовка.

### Как это связано с проектом

Архитектура отражена в структуре каталогов проекта и документации. Основной поток данных реализован как `Frontend -> Backend API -> Benchmark Engine -> PostgreSQL`, а real benchmark дополнительно использует связку `Backend -> k6 -> benchmark-target` и `Backend -> Docker CLI -> Docker daemon`.

### Как это связано с НИР1/2/3

Этап продолжает НИР1 за счет прикладного использования микросервисной архитектуры, контейнеризации и DevOps-подхода. С НИР2 он связан выбором метрик для оценки инфраструктурных решений. С НИР3 он связан сохранением сравнительной логики VM vs Docker и переходом от экспериментального сценария к платформенной реализации.

## Этап 2

### Название этапа

Разработка backend API и модели хранения benchmark-запусков.

### Суть работ

На втором этапе реализован backend на NestJS, отвечающий за прием запросов, запуск benchmark engine, управление статусами выполнения и сохранение результатов в PostgreSQL. Для benchmark-запусков предусмотрены статусы `pending`, `running`, `completed`, `failed`, режимы `demo` и `real`, а также типы окружения `vm` и `docker`.

Модель данных включает метрики производительности и эксплуатационные показатели: RPS, latency, p95 latency, error rate, total requests, startup time, restart time, scaling time, lead time, time to provision и CPU overhead.

### Как это связано с проектом

Backend предоставляет API:

- `GET /health`;
- `GET /benchmark-runs`;
- `GET /benchmark-runs/:id`;
- `POST /benchmark-runs`.

Результаты сохраняются в таблице `benchmark_runs` через TypeORM и PostgreSQL. API используется frontend-dashboard и CI/CD smoke-тестом.

### Как это связано с НИР1/2/3

Связь с НИР1 проявляется в реализации отдельного backend-сервиса как части микросервисной системы. Связь с НИР2 выражена в переносе критериев оценки в структуру данных benchmark-запуска. Связь с НИР3 обеспечивается сохранением набора экспериментальных метрик и поддержкой сравнения VM vs Docker.

## Этап 3

### Название этапа

Реализация benchmark engine с демонстрационным и реальным режимами измерений.

### Суть работ

На третьем этапе реализован benchmark engine со стратегиями выполнения. Demo mode генерирует реалистичные учебные значения метрик для быстрой демонстрации и проверки интерфейса. Real mode запускает k6 against target service, извлекает RPS, среднюю latency, p95 latency, error rate и total requests.

Для Docker-сценария дополнительно реализовано измерение runtime-показателей через Docker CLI: startup time, restart time и scaling time. Если Docker CLI недоступен, benchmark может завершиться с предупреждением в `notes`.

### Как это связано с проектом

Benchmark engine реализован через отдельные сервисы `DemoBenchmarkService`, `RealBenchmarkService` и `DockerRuntimeMetricsService`. Для real benchmark используется отдельный `benchmark-target` сервис с endpoint для разных типов нагрузки: health, CPU, memory, random delay и large JSON payload.

### Как это связано с НИР1/2/3

Этап продолжает НИР2 через практическую реализацию измеримых критериев, а НИР3 через воспроизведение benchmark-нагрузки и сбор метрик производительности. Связь с НИР1 проявляется в сравнении контейнерного сценария с VM baseline и использовании Docker как объекта исследования.

### Какие артефакты подтверждают выполнение

- `backend/src/benchmark-runs/demo-benchmark.service.ts`;

- `benchmark-result.json`.

## Этап 4

### Название этапа

Разработка frontend-dashboard для визуализации и интерпретации benchmark-результатов.

### Суть работ

На четвертом этапе реализован frontend на React, TypeScript и Vite. Dashboard позволяет выбрать benchmark-режим, тип окружения, target service или custom URL, задать параметры real benchmark и запустить benchmark через backend API.

Интерфейс отображает карточки метрик, графики RPS и latency, сравнение времени масштабирования, таблицу VM vs Docker, исторический тренд, recommendation panel и таблицу истории запусков.

### Как это связано с проектом

Frontend является пользовательским уровнем платформы и обеспечивает наглядную интерпретацию результатов. Он получает данные из `GET /benchmark-runs`, запускает новые benchmark через `POST /benchmark-runs` и показывает сохраненную историю из PostgreSQL.

### Как это связано с НИР1/2/3

Этап связан с НИР2 через визуализацию выбранных критериев оценки и с НИР3 через отображение результатов VM vs Docker экспериментов. Он усиливает практическую применимость результатов НИР1, так как показывает влияние контейнеризации в форме, удобной для анализа и защиты.

## Этап 5

### Название этапа

Контейнеризация, локальная оркестрация и подготовка DevOps-контура.

### Суть работ

На пятом этапе компоненты проекта были упакованы в Docker-образы и объединены через Docker Compose. Локальная инфраструктура включает PostgreSQL, backend, frontend и benchmark-target. Для backend-контейнера предусмотрен доступ к Docker socket, что необходимо для измерения Docker runtime metrics.

Также добавлен CI workflow, который собирает frontend и backend, собирает Docker images, запускает часть сервисов через Docker Compose, выполняет real benchmark smoke test и сохраняет результат как artifact.

Terraform-слой представлен как минимальная учебная IaC-заготовка. Он не создает реальные cloud-ресурсы, но фиксирует направление дальнейшего развития инфраструктурного слоя.

### Как это связано с проектом

Docker Compose обеспечивает воспроизводимый локальный запуск всей платформы. GitHub Actions проверяет, что проект собирается и способен выполнить базовый real benchmark в автоматическом окружении. Terraform-каталог показывает подготовку к инфраструктуре как коду.

### Как это связано с НИР1/2/3

Этап напрямую продолжает НИР1 через практическое применение контейнеризации и DevOps-процессов. Он связан с НИР2 через снижение эксплуатационной сложности и повышение воспроизводимости проверок. С НИР3 он связан использованием инфраструктурного слоя, benchmark-нагрузки и автоматизированного запуска эксперимента.

### Название этапа

Документирование, проверка результатов и подготовка материалов для итогового отчета.

### Суть работ

На шестом этапе подготовлена документация, описывающая архитектуру, API, методологию benchmark, CI/CD, локальный запуск и ручную проверку перед демонстрацией. Также сформирован адаптированный план практики, согласующий фактическую реализацию проекта с темой ВКР и результатами НИР1, НИР2 и НИР3.

Отдельно зафиксированы ограничения: demo mode является симуляцией, real mode зависит от доступности k6, Docker runtime metrics требуют Docker CLI и доступа к Docker daemon, Terraform пока является заготовкой, а не завершенным cloud deployment.

Документация делает проект воспроизводимым для запуска, проверки и демонстрации. Она также помогает корректно интерпретировать результаты benchmark и не смешивать реальные измерения с демонстрационными метриками.

## Почему план полностью соответствует теме ВКР

План соответствует теме ВКР, поскольку вся практическая работа направлена на исследование влияния контейнеризации на развертывание, масштабируемость и производительность микросервисной архитектуры.

Проект позволяет сравнивать VM baseline и Docker по набору метрик, связанных с эффективностью развертывания и масштабирования. Контейнеризация рассматривается не только как способ упаковки приложения, но и как часть DevOps-процесса: локальная оркестрация, автоматическая сборка, benchmark smoke test, сохранение истории и визуализация результатов.

План также честно отражает границы текущей реализации. Реально измеряемыми являются k6-метрики HTTP-нагрузки и Docker runtime metrics. Demo mode и часть эксплуатационных показателей используются как учебная модель для демонстрации методологии. Такой подход делает результат пригодным для итогового отчета, так как он показывает не только разработку программного продукта, но и исследовательскую логику оценки инфраструктурных решений.

## Почему такой план сильнее исходного шаблона

Адаптированный план сильнее исходного шаблона по следующим причинам:

- он строится вокруг фактически реализованной benchmark-платформы, а не вокруг абстрактного набора работ;
- он не требует менять проект под старые формулировки, а корректно описывает достигнутый результат;
- он связывает каждый этап с конкретными файлами, компонентами и проверяемыми артефактами;
- он разделяет реальные измерения, демонстрационные метрики и ограничения методологии;
- он усиливает связь с ВКР через DevOps, контейнеризацию, масштабирование, benchmark-нагрузку и CI/CD;
- он логично продолжает НИР1, НИР2 и НИР3: от теории, через критерии, к эксперименту и затем к автоматизированной платформе;
- он удобен для итогового отчета, потому что этапы уже соответствуют разделам, которые можно раскрывать в тексте отчета.
